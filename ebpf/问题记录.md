## invalid indirect read from stack

写ebpf程序的时候，遇到了"invalid indirect read from stack"这个问题，详细信息如下：

~~~
panic: program sys_enter_sendmmsg: load program: permission denied: invalid indirect read from stack off -80+12 size 16 (529 line(s) omitted)
~~~

从字面意思看是间接读取某个栈的位置无效，给出的错误信息只包含是在哪个函数触发的，没有具体到是哪一行代码。首先想到如何确定是哪一行代码导致了这个问题，以及导致这个问题的原因。

确定是哪一行代码有两种思路:

* 比较笨的方法，通过注释代码的方法确定
* 反汇编 llvm-objdump。

通过反汇编的方法并没有找到使用"-80 + 12"的代码片段，所以就使用了方法1，发现是调用bpf_map_update_elem时触发的问题。

为什么bpf_map_update_elem会导致这个问题呢，开始走了不少弯路，直到发现了https://www.kernel.org/doc/html/v5.17/bpf/verifier.html这篇文档，上面记录了一些ebpf verifier校验不通过及原因的代码样例，其中就有"invalid indirect read from stack"的错误样例。结合样例和代码的反汇编大致确定了问题所在

导致上述问题的原因:

结构体write_msg_ctx 实际占用了16字节的栈空间(内存对齐)，在栈上的起始地址是r10 - 80(结束地址是r10-64)。因为write_msg_ctx.len是int类型（4字节)，给其赋值时 的"63 1a b8 ff 00 00 00 00 *(u32 *)(r10 - 72) = r1 " 代码只初始化了r10 - 72 开始的4字节，r10-68 开始的4字节并未初始化(即报错信息中的-80 + 12的位置)，所以在访问r10-68这块空间时就会出现"invalid indirect read from stack"

程序片段

~~~c
struct write_msg_ctx{
    void *data;
    int len;
};

SEC("tracepoint/syscalls/sys_enter_sendmmsg")
int static inline sys_enter_sendmmsg(struct enter_sendmmsg_arg *ctx){
    if(!is_valid_pid()){
        return 0;
    }
   struct mmsghdr *hdr = ctx->msgvec;
    struct write_msg_ctx write_msg = {
        .data = hdr,
        .len = ctx->vlen ,
    };
    handle_socket_write_enter( ctx, ctx->fd, &write_msg);
   return 0;
}


static int  __always_inline handle_socket_write_enter( void *ctx, int fd, void *key_value){
    //.......
    struct write_arg_key_v1 key = {
        .tgid_pid = tgid_pid,
    };
    int ret = bpf_map_update_elem(&write_msgs_map, &key, key_value, BPF_ANY);
    if(ret < 0){
        bpf_printk("map_update_elem failed. %s ret:%d\n", comm, ret);
    }
    //.............
}
~~~

将结构体write_msg_ctx改成如下，成功解决上述问题。

~~~c
struct write_msg_ctx{
    void *data;
    int len;
};
~~~
