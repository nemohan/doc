## seek

实现zorodb的过程中，写入key-value时，直接写入到文件末尾位置，而使用seek来实现key-value随机读取。后面意识到seek会对写入产生影响，因为seek会改变文件当前的读写位置，而write-read-write这样的序列，可能会导致read之后的write不是写入文件末尾，而是覆盖了上次位于文件读取位置的key-value。经验证确实如此。而在zorodb中验证这个问题时，确始终没有触发这个问题。这就有点自相矛盾了。仔细阅读实现随机读取的代码，发现问题出在bufio.NewReader，为啥问题出在这呢？因为测试时，读者只写入了几个长度只有几字节的key-value，而bufio.NewReader创建的reader的默认buffer为4KB。这样就导致write-read-write这样的操作序列中的read实际读取了4KB内容（已经超过文件末尾)，文件位置重新定位到了文件末端，之后的write操作就不会覆盖之前的内容。若不使用bufio读取，就会产生覆盖

~~~go
	s.file.Seek(int64(valuePos), io.SeekStart)
	crc := uint32(0)
	buf := bufio.NewReader(s.file)
	binary.Read(buf, binary.BigEndian, &crc)
    //....其他代码
~~~



## mmap

mmap可以将磁盘上的文件映射到内存。可以不使用read、write而以直接操纵内存的方式，读写文件内容。

mmap函数原型：

~~~c
#include <sys/mman.h>
void *mmap(void *addr, size_t len, int prot, int flag, int fd, off_t off);
~~~

参数：

* addr:  指定映射区域的起始地址。通常指定为0，让系统选择合适的地址。mmap的返回值是内存区域映射的起始地址
* fd: 文件描述符（必须是已经打开的文件)
* len: 指定映射区域的大小
* off: 映射区域对应的文件中的起始位置，通常要求off和addr参数是 虚拟内存页面大小(virtual memory page size)的整数倍。
* prot: 指定映射区域的保护标志，可以是PROT_NONE或 PROT_READ、PROT_WRITE、PROT_EXEC的or组合。<font color="red">需要注意的是映射区域的权限不能超过文件打开时指定的权限，如以read-only打开的文件不能指定PROT_WRITE权限</font>

| prot 标志  | 描述               |
| ---------- | ------------------ |
| PROT_READ  | 可读               |
| PROT_WRITE | 可写               |
| PROT_EXEC  | 可执行             |
| PROT_NONE  | 映射区域不能被访问 |

* flag: 

  | flag        | 描述                                                         |
  | ----------- | ------------------------------------------------------------ |
  | MAP_FIXED   | mmap返回的地址必须是addr参数指定的地址（不推荐使用此参数）   |
  | MAP_PRIVATE | 对映射区域的任何写入操作都会导致创建映射文件的一份拷贝，写入是写入拷贝中。后续对文件内容的引用，都是引用拷贝中的内容。（**原始文件不受写入影响**) |
  | MAP_SHARED  | 指定对映射区域的写入操作等效于写入文件。不能和MAP_PRIVATE同时使用 |



### munmap

映射区域在程序结束时会自动解除映射，或者我们可以调用munmap解除映射。**映射后关闭文件描述符不影响映射。**

<font color="red">munmap的调用不会导致映射区域的内容写回磁盘文件。由MAP_SHARED标志保护的映射区域对应的磁盘文件的更新由内核的虚拟内存算法决定。对由MAP_PRIVATE标志保护的映射区域的更改会在解除映射时自动丢弃.</font>

~~~c
#include <sys/mman.h>
int munmap(void *addr, size_t len)
~~~



### 使用时注意事项

#### 和mmap相关的两个信号量

* SIGSEGV: 通常指明访问的内村对我们不可用，写入设置了read-only标志的映射区域也会导致产生这个信号量
* SIGBUS:  引用无效的内存映射区域。比如引用映射内存区域之前，其他进程将文件大小截短(truncated)。此时再引用截断之后的部分就会导致产生该信号量

#### 磁盘文件同步

<font color="red">当写入用MAP_SHARED标志设置的映射区域时，数据并不会立即写入磁盘文件。内核会根据a)系统负载 b)由于系统故障导致数据丢失的限制的配置参数(将数据丢失限定在指定范围内)，确定何时将脏页面(dirty page)写回磁盘。数据改变写回磁盘的最小单元是页面(page)，即改变了一个字节，也会导致整个页面被写回</font>

如果映射区域的内容被改变，可以调用msync将改变的内容写回磁盘文件。msync类似fsync函数。**如果映射区域为私有(即设置了MAP_PRIVATE标志)，磁盘文件不会被改变**。addr参数需要对齐到页面(page boundary)

~~~c
#include <sys/mman.h>
int msync(void *addr, size_t len, int flags)
~~~

参数:

* addr: 指定写回磁盘文件的映射区域地址
* flag: MS_ASYNC标志只是简单的调度将被写回的页面，MS_SYNC（同步）等待数据写回磁盘完成， MS_INVALIDATE 通知系统丢弃和磁盘文件不一致的页面(page)，某些实现会丢弃指定区域的所有页面(指定区域是len范围内的么？)，此行为不要求

以下是mmap的*nix系统实现:

~~~go
//+build !windows,linux
package fileutil

import (
	"os"

	"golang.org/x/sys/unix"
)

const (
	MS_SYNC  = unix.MS_SYNC
	MS_ASYNC = unix.MS_ASYNC
)

func mmap(file *os.File, size int) ([]byte, error) {
    //若需要测试mmap写入性能，需要将unix.PROT_READ 修改为 unix.PROT_WRITE
    //同时，unix.MAP_PRIVATE替换成unix.MAP_SHARED
	data, err := unix.Mmap(int(file.Fd()), 0, size, unix.PROT_READ, unix.MAP_PRIVATE)
	if err != nil {
		return nil, err
	}
	return data, nil
}

func msync(data []byte, flag int) error {
	return unix.Msync(data, flag)
}
func munmap(data []byte) error {
	return unix.Munmap(data)
}

~~~
